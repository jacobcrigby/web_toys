Marble Game — Alpha Plan

A tilt-to-roll 3D marble game in the lineage of Labyrinth, Perplexus, and Rollgoal. The player never moves the marble directly — they tilt the surface and gravity does the rest. This document specifies the Alpha: a glassy marble rolling on a tiltable table with walls and collision-test boxes.

1 · Alpha goal & scope

The Alpha exists to prove the foundations every later feature stands on: rendering, physics, tilt control, and the audio seam. It is a physics sandbox, not a game with objectives yet.

In scope

  • Glassy, smoky, see-through marble
  • Flat table + raised walls, tilts on two axes
  • Tilt via keyboard, mouse, gamepad
  • Dynamic test objects: boxes, ramps, cylinders
  • Fixed angled overhead camera
  • Reset (re-center marble, level table)
  • Debug overlay (FPS + Inspector toggle)
  • Audio seam wired to a silent stub

Out of scope

  • Goal hole / win / lose states
  • Levels, mazes, level loading
  • Real audio assets, music
  • Menus, scoring, persistence
  • Inside-a-sphere (Perplexus) mode
  • Device tilt / gyro controls (deferred)
  • Multiple marbles

Why so small? KISS and YAGNI. A goal hole or a maze is cheap to add after the marble rolls and collides reliably, and expensive to debug if bolted on before the physics feels right. The Alpha de-risks the hard part.

3 · Architecture

Layered, one concern per module. Nothing reaches across layers: input does not touch physics, the table does not read the keyboard, lifecycle code does not build materials.

marblegame/
  index.html                 Vite entry, mounts <canvas>
  src/
    main.ts                  bootstrap: canvas + GameApp, start loop
    config/
      GameConfig.ts          single source of truth for every tunable
    core/
      GameApp.ts             owns Engine, Scene, render loop, resize, lifecycle
    physics/
      PhysicsWorld.ts        Havok init (await wasm), gravity, enable physics
    scene/
      SceneBuilder.ts        fixed camera, lights, environment / IBL
      Table.ts               tiltable surface + walls (ANIMATED) under a pivot node
      Marble.ts              glassy smoky sphere + DYNAMIC body
      TestObjects.ts         DYNAMIC test bodies: boxes, ramps, cylinders
    input/
      TiltIntent.ts          { x, z } normalized to [-1, 1]
      InputSource.ts         interface: read() -> TiltIntent, dispose()
      InputManager.ts        aggregates sources, combines + clamps, drives Table
      KeyboardInput.ts       baseline
      MouseInput.ts          pointer drag
      GamepadInput.ts        analog stick
    audio/
      AudioService.ts        interface + NullAudioService; collision hook
    ui/
      DebugOverlay.ts        FPS readout + Inspector toggle

Rule of Two in practice: the three input sources justify the InputSource interface immediately — there is more than one real implementation on day one. Do not pre-build abstractions that have a single implementation (e.g. no "SurfaceProvider" interface for a table that only exists in one form yet).

4 · Tilt data flow

Input is unidirectional. Each source emits a normalized intent; the manager sums and clamps them; the table converts intent into a pivot rotation; Havok and gravity move the marble.

┌──────────────┐     ┌───────────┐     ┌────────────┐
│ KeyboardInput│     │ MouseInput│     │GamepadInput│
└──────┬───────┘     └─────┬─────┘     └─────┬──────┘
       │  TiltIntent       │                 │
       └─────────┬─────────┴─────────────────┘
                 ▼
        ┌───────────────┐   sum + clamp to maxTiltAngle
        │  InputManager  │ ───────────────────────────────┐
        └───────────────┘                                 ▼
                                              ┌──────────────────────┐
                                              │ Table.applyTilt(x,z)  │
                                              │  rotates pivot node    │
                                              └───────────┬───────────┘
                                                          ▼
                              Havok: table/walls ANIMATED, marble DYNAMIC
                                                          ▼
                                  gravity rolls the marble downhill
                                                          ▼
                              collision observable → AudioService.playClack()

A TiltIntent is just { x: number; z: number } where each component is in [-1, 1]. x tips the table forward/back, z tips it left/right. Sources never know about the table, physics, or each other.

6 · Physics model

  • Marble & boxes — DYNAMIC. Mass, friction, and restitution come from GameConfig. The marble uses a sphere collider; boxes use box colliders.
  • Table & walls — ANIMATED (kinematic). They are children of a single pivot TransformNode. Rotating the pivot moves all of them together and correctly transfers momentum to dynamic bodies — which a static body would not do.
  • Ground / environment — static. Not strictly required once the table exists, but useful as a reference plane during early phases.
  • Init order matters. Havok's wasm must be awaited before any body is created. GameApp bootstraps physics before scene content.
// Conceptual init order (not final code)
const havok = await HavokPhysics();
scene.enablePhysics(new Vector3(0, GameConfig.gravityY, 0),
                    new HavokPlugin(true, havok));
// ...then build table (ANIMATED), marble (DYNAMIC), boxes (DYNAMIC)

7 · The glassy marble

The default marble is a glassy, smoky, see-through sphere. Implemented with a Babylon PBRMaterial:

  • Low roughness, metallic 0 — a clean glass surface.
  • Refraction / transmission enabled (subSurface) so the scene shows through the marble.
  • A smoky tint via a dark, desaturated volume/tint color plus partial alpha, so it reads as smoked glass rather than clear glass.
  • An environment (IBL) texture so reflections and refraction look like real glass. SceneBuilder provides it; the marble consumes it.

Performance note: transmission/refraction is the most expensive material in the scene. There is exactly one marble, which keeps this cheap. Revisit on mobile during the tuning pass; a cheaper fallback material is a later concern, not an Alpha one.

8 · Input system

Every device implements the same tiny contract, so adding or removing a source changes nothing downstream.

interface InputSource {
  // Current tilt request, each axis in [-1, 1].
  read(): TiltIntent;
  dispose(): void;
}
  • KeyboardInput — arrows / WASD. The always-available baseline; the Alpha is playable with nothing else wired.
  • MouseInput — pointer drag from center maps to tilt. Requires the camera's user-controls to be detached so it owns the pointer.
  • GamepadInput — left analog stick via the Gamepad API; deadzone from config.

InputManager reads all active sources each frame, sums their intents, clamps the result, and calls Table.applyTilt. Summing keeps multi-device behavior predictable; clamping prevents over-tilt.

Deferred: device tilt / gyro (deviceorientation) is out of the Alpha — it adds iOS permission prompts, a user-gesture requirement, and an HTTPS dependency. Because every device is just an InputSource, adding it later is a single new file with no downstream changes.

9 · Audio seam

The Alpha ships silent but ready. We build the contract and the collision hook now so wiring real clacks later is a one-file change.

interface AudioService {
  // volume in [0, 1], typically derived from impact magnitude
  playClack(volume: number): void;
}
class NullAudioService implements AudioService {
  playClack(): void { /* intentionally silent in Alpha */ }
}

A throttled collision observable maps impact magnitude to a volume and calls playClack. In Alpha the wired implementation is NullAudioService; later we swap in a clip- or synth-backed one without touching the collision code.

11 · Build phases

Plan-driven: build top to bottom, one phase at a time. A phase is done only when its acceptance criterion holds. PLAN.md holds the live checklist; this is the narrative.

0Scaffolding

Vite + TypeScript + Babylon + Havok dependencies; tsconfig, vite.config, index.html with a fullscreen canvas; main.ts renders an empty scene; npm scripts.

Accept: npm run dev shows a blank rendered scene; typecheck clean; no console errors.

1Scene foundation

GameApp lifecycle + render loop + resize. SceneBuilder adds the fixed angled camera (controls detached), lighting, and an IBL environment, plus a reference ground.

Accept: a lit scene renders from the fixed camera; resize handled.

2Physics world

PhysicsWorld awaits Havok, enables physics with gravity. A temporary dynamic sphere falls and rests on the ground.

Accept: sphere falls and settles; stable, no jitter/explosion.

3The marble

Marble builds the glassy smoky PBR sphere with a dynamic body tuned from config; spawns above the surface and settles.

Accept: marble reads as see-through smoked glass and rolls/settles believably.

4Tilting table

Table builds surface + four walls as ANIMATED bodies under a pivot node; applyTilt rotates within tilt limits. A temporary hard-coded tilt proves rolling.

Accept: tilting rolls the marble downhill; walls contain it.

5Input system

TiltIntent + InputSource + InputManager; keyboard first, then mouse, gamepad. Intent drives the table.

Accept: keyboard tilts; each other source tilts when present; combined input is clamped sanely.

6Test objects

TestObjects spawns a few dynamic bodies on the table — boxes, ramps (inclined wedges), and cylinders — each with a correct collider shape to exercise gravity feel and non-square contact.

Accept: the marble knocks objects around, rolls believably down ramps and along cylinders, and everything respects the walls and rests realistically.

7Audio seam

AudioService + NullAudioService; throttled collision hook maps impact magnitude to playClack(volume).

Accept: collisions call the stub with a sensible volume; zero errors; silent.

8Debug & reset

DebugOverlay shows FPS and toggles the Babylon Inspector; a reset key re-centers the marble (clears velocity) and levels the table.

Accept: FPS visible, Inspector toggles, reset returns to a clean start.

9Tuning pass

Tune gravity, friction, restitution, tilt sensitivity, and the marble material in GameConfig.

Accept: rolling feels good and controllable; stable ~60 FPS on desktop.

12 · Risks & watch-items

Havok wasm not loaded before bodies are createdAwait HavokPhysics() in bootstrap before any scene content.
Table as static body doesn't transfer momentumTable/walls are ANIMATED, driven by the pivot transform.
Ramps/cylinders using box collidersUse a wedge/convex collider for ramps and a cylinder collider for cylinders, or roll feel will be wrong.
Glass material cost on mobileSingle marble; revisit a fallback in tuning, not before.
Mouse-tilt fights the cameraDetach camera user-controls so the pointer is free for tilt.
Physics jitter from over-tiltClamp TiltIntent and cap maxTiltAngle.

13 · Engineering principles

Carried into every phase and enforced in review. The full operational version lives in AGENTS.md.

  • KISS — the simplest thing that meets the acceptance criterion.
  • YAGNI — build only what the current phase needs.
  • DRY with the Rule of Two — abstract on the second real use, never the first. Premature abstraction is a defect.
  • Single responsibility — small modules, one clear job each.
  • Comments describe the present. How the code works now — history belongs in Git. No narrative, no restating code, no commented-out blocks. Prefer a clear name over a comment.
  • Plan stays in sync. Update PLAN.md (and this doc) in the same change that alters the design.
